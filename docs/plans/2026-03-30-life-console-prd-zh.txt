Life Console 产品需求文档（PRD）
版本：V1
日期：2026-03-30
平台：iOS

1. 产品定义

产品名：Life Console
一句话定义：一个帮助用户守住生活底线、推动关键产出、并通过 AI 教练与真实反馈找回掌控感的个人控制台。
产品类型：习惯系统 + 项目推进系统
初始用户：先为创始人自己使用，后续视验证结果考虑商业化

2. 背景与问题

用户并不缺目标，也并非不知道什么重要。真正的问题是：
- 先是节奏失控：作息、精力、日常秩序不稳定
- 然后执行失控：知道该做什么，但拖延、破功、开始不了
- 最后是自我信任下降：怀疑自己是否真的在变好

市面上很多习惯 App 更像记录器，而不是接管系统。它们擅长记录“有没有做”，但不擅长在用户即将失速时及时介入、解释失败原因、重新拉回轨道。

本产品的核心不是“帮用户打卡”，而是“让用户每天结束时能明确感受到：我守住了底线，也推动了真正重要的事”。

3. 产品目标

3.1 核心目标
1. 显著提升用户每天对重要事情的可见推进，特别是 side project 的实际产出。
2. 稳定少数高杠杆习惯，作为执行力的能量底盘。
3. 通过清晰优先级、诚实记录和 AI 诊断，重建掌控感与自我信任。

3.2 非目标
- 不是社交型习惯社区
- 不是泛 To-do List 替代品
- 不是偏疗愈或冥想导向的 wellness 产品
- 不是儿童化游戏化打卡工具

4. 目标用户与关键洞察

当前目标用户就是创始人本人，对自己的需求已经非常清楚：
- 想要的是“人生控制台”，不是简单 habit tracker
- 每天首页必须同时存在两条主线：
  - 1 个关键习惯：守住底线
  - 1 个关键项目动作：推动产出
- 对少数关键项希望系统足够强硬，不只是温柔鼓励
- 希望提醒是“事前提醒 + 逾期追踪”双机制，而不是只在开始前叫一下
- AI 应该具备拆解、诊断、督促三种能力，而不是生成空泛建议

4.1 隐性需求
- 不想再被“我今天很忙”这种伪进展欺骗，所以 side project 必须按可见产出来定义完成。
- 日记不是装饰型功能，而是情绪排毒和执行复盘的关键入口。
- 日记应分两层：
  - App 内 daily journaling
  - 隔天一次的纸质日记，由 App 监督完成

5. 产品原则

1. 先提供清晰感，再提供激励感。
2. 优先真实记录，不制造虚假的完成感。
3. 高压机制只用于少数高杠杆行为。
4. 对重要项目，结果比投入时间更重要。
5. 日常使用要能在坏状态下仍然坚持。
6. AI 必须能解释失败，而不只是安慰失败。

6. 成功指标

6.1 结果指标

A. 核心结果指标一：项目推进结果
- 每周可见产出天数
  - 定义：一周内有多少天产生了可见 side project 产出
  - 合法产出示例：代码提交、原型页面、PRD 模块、已验证迭代、已交付文案
  - 计算公式：一周内“至少有 1 个有效可见产出”的自然日数量
  - 观察周期：每周
  - 预警阈值：
    - 绿：5-7 天
    - 黄：3-4 天
    - 红：0-2 天
  - Dashboard 展示：
    - 7 天条形图
    - 当周产出物列表
- 周里程碑推进率
  - 定义：这一周是否至少推进了一个有意义的项目里程碑
  - 里程碑示例：新页面、新模块、onboarding 节点、功能迭代、产品文档关键章节
  - 计算公式：有里程碑推进的周数 / 总观察周数
  - 观察周期：每周
  - 预警阈值：
    - 绿：>=75%
    - 黄：50-74%
    - 红：<50%
  - Dashboard 展示：
    - 本周里程碑状态卡
    - 4 周滚动趋势
- 关键任务声明后完成率
  - 定义：一旦声明了“今日关键项目动作”，最终在当天完成的比例
  - 计算公式：当天完成的已声明关键任务数 / 当天声明关键任务总数
  - 观察周期：每周
  - 预警阈值：
    - 绿：>=80%
    - 黄：60-79%
    - 红：<60%
  - Dashboard 展示：
    - 比例卡片
    - 已声明 / 已完成 / 未完成明细列表

B. 核心结果指标二：生活节律稳定度
- 关键基础习惯综合稳定度
  - 定义：对核心基础习惯做动态加权后的稳定度得分
  - 核心纳入：
    - 早睡
    - 阅读
    - App 日记
    - 纸质日记
  - 观察纳入：
    - 听播客
    - 锻炼或慢走 13000 步
    - 法语
    - 每周打扫一次家里卫生
  - 动态权重规则：
    - 第一优先：对 side project 影响最大
    - 第二优先：用户当前手动标记最关键
    - 第三优先：最近连续失守
  - 计算公式：
    - 单习惯周得分 = 准时完成 * 1.0 + 最低完成 * 0.6 + 补做完成 * 0.4
    - 综合稳定度 = sum(单习惯得分 * 动态权重) / sum(满分 * 动态权重) * 100
  - 观察周期：每周
  - 预警阈值：
    - 绿：>=85
    - 黄：70-84
    - 红：<70
  - Dashboard 展示：
    - 总稳定度主卡
    - 分习惯权重拆解
    - 核心纳入 vs 观察纳入分层显示
- 晚间收尾稳定度
  - 定义：每周有多少天按时进入晚间收尾与睡前流程
  - 计算公式：按时进入晚间收尾流程的天数 / 计划天数
  - 观察周期：每周
  - 预警阈值：
    - 绿：>=80%
    - 黄：60-79%
    - 红：<60%
  - Dashboard 展示：
    - 晚间时间轴条带
    - 7 天稳定性折线 / 火花线

6.2 诊断辅助指标
- 掌控感周评分
  - 计算公式：用户每周自评 1-5 分，系统保留最近 4 周滚动均值
  - 用途：帮助解释波动，但不作为核心成功标准
  - Dashboard 展示：
    - 与客观指标并排的趋势线
- 主线跑偏检查
  - 题目：今天我做的最重要的事，是否仍然是今天最该做的事？
  - 计算公式：回答“是”的天数 / 有回答的天数
  - 用途：帮助判断决策清晰度与执行偏移
  - 预警阈值：
    - 绿：>=80%
    - 黄：60-79%
    - 红：<60%
  - Dashboard 展示：
    - 每日 Yes/No 条带
    - 每周主线偏移摘要

6.3 行为指标 / Product Health

A. 执行转化率
- 全链路执行转化率
  - 定义：从“提醒 -> 开始 -> 完成 -> 产出”全链路的转化情况
  - 计算公式：
    - 开始率 = 已开始 / 已提醒
    - 完成率 = 已完成 / 已开始
    - 产出率 = 有可见产出 / 已完成的可产出任务
    - 全链路分数 = 三段转化率的加权综合
  - 观察周期：日 / 周
  - 预警阈值：
    - 绿：>=75
    - 黄：55-74
    - 红：<55
  - Dashboard 展示：
    - 漏斗图
    - Hard / Soft 项目分层漏斗
  - 用途：第一 Product Health 指标
- 关键任务声明后完成率
  - 定义：当天声明的关键任务，最终按时完成的比例
  - 计算公式：按时完成的已声明关键任务数 / 已声明关键任务总数
  - 观察周期：每周
  - Dashboard 展示：
    - 周比例卡
    - 与全链路执行转化率并列
  - 用途：第二 Product Health 指标

B. 决策清晰度
- 关键项确定速度
  - 定义：用户打开 App 后，确认今日关键习惯和关键任务所花的时间
  - 计算公式：从打开 App 到确认两个关键项的中位秒数
  - 观察周期：日 / 周
  - 预警阈值：
    - 绿：<=90 秒
    - 黄：91-180 秒
    - 红：>180 秒
  - Dashboard 展示：
    - 中位耗时卡片
    - 7 日 / 30 日趋势
  - 用途：第一决策清晰度指标
- 主线未跑偏率
  - 定义：晚上回看时，用户确认当天实际投入仍围绕既定主线推进的比例
  - 计算公式：回答“未跑偏”的天数 / 有回答的天数
  - 观察周期：每周
  - 预警阈值：
    - 绿：>=80%
    - 黄：60-79%
    - 红：<60%
  - Dashboard 展示：
    - 周度主线一致率
    - 与关键任务完成率配对展示
  - 用途：第二决策清晰度指标

6.4 指标采集原则
- 默认以系统自动计算为主
- 少量主观确认作为补充
- 适合主观确认的内容包括：
  - 掌控感周评分
  - 主线跑偏检查
  - 日终是否仍认可今日主线

7. 核心对象模型

7.1 Habit（习惯）
具有周期、完成标准、提醒规则和强弱执行策略的重复行为。
字段建议：标题、分类、周期、标准完成、最低完成、证据类型、提醒方案、Hard/Soft 级别、补做逻辑。

7.2 Mission（任务 / 项目动作）
与长期项目推进强绑定、强调“可见产出”的高优先级动作。
字段建议：任务名、输出目标、截止时间、完成证据、状态、权重。

7.3 Journal（日记）
分为两类：
- Daily in-app journal：App 内每日短复盘
- Paper journal supervision：隔天纸质日记监督

7.4 Review（复盘）
用于日度和周度总结，承接 AI 诊断、趋势分析和策略修正。

8. V1 习惯清单

- 23:00 前早睡
  - 周期：每日
  - 标准完成：23:00 前关灯入睡
  - 最低完成：22:40 前进入睡前流程
  - 执行等级：Hard

- 听播客
  - 周期：每日
  - 标准完成：收听 20 分钟以上，主题为 AI / Tech / Investing / Consumer Goods / Healthcare / 出海
  - 最低完成：收听 10 分钟以上
  - 执行等级：Hard

- 锻炼 30 分钟 / 慢走 13000 步
  - 周期：每日
  - 标准完成：慢走 / 逛街 / 瑜伽 / 慢跑等累计 30 分钟，或完成 13000 步
  - 最低完成：15 分钟轻运动
  - 执行等级：Soft

- 阅读
  - 周期：每日
  - 标准完成：1 章纸质书 + 15 分钟电子书
  - 最低完成：完成其一
  - 执行等级：Hard

- 法语学习
  - 周期：每日
  - 标准完成：完成 1 节课程
  - 最低完成：10 分钟主动练习
  - 执行等级：Soft

- AI 工具推进 side project
  - 周期：每日
  - 标准完成：产出可见成果：代码、原型、PRD、文案或已验证迭代
  - 最低完成：默认不设置轻量最低标准
  - 执行等级：Hard

- App 内日记
  - 周期：每日
  - 标准完成：完成 guided journal
  - 最低完成：3 句话快速记录
  - 执行等级：Soft

- 纸质日记
  - 周期：隔天一次
  - 标准完成：完成纸写并在 App 内确认
  - 最低完成：无
  - 执行等级：Hard

- 每周打扫一次家里卫生
  - 周期：每周
  - 标准完成：完成一次有明确范围的居家清洁
  - 最低完成：完成一个主要区域的集中清洁
  - 执行等级：Soft

9. 产品范围

9.1 V1 MVP
1. AI 协助 onboarding，把目标转成可执行规则
2. Today 首页，展示 1 个关键习惯 + 1 个关键项目动作
3. 提醒系统：事前提醒、逾期追踪
4. 完成状态记录：完成 / 最低完成 / 失败 / 补做
5. App 内 daily journaling
6. 隔天纸质日记监督
7. AI 教练：设置、失败诊断、日终反馈
8. Progress 看板与周复盘
9. 基础游戏化分数系统

9.2 V1.5
- 更智能的推荐与节律调整
- 失败模式聚类
- 按时段与周几做动态建议
- 更完整的趋势可视化

9.3 V2
- 社交监督或搭子机制
- 习惯模板 / 计划模板市场
- 商业化验证模块

10. 关键用户流程

10.1 首次设置
1. 用户用自然语言描述目标。
2. AI 自动拆解出习惯、任务、提醒时间和完成标准。
3. 用户确认哪些是 Hard，哪些是 Soft。
4. 系统生成首页控制台和提醒计划。

10.2 日常使用
1. 打开 Today 首页，3 秒内看清今日最重要的两件事。
2. 接收事前提醒并开始执行。
3. 若逾期未完成，系统按 Hard/Soft 级别触发追踪提醒。
4. 完成后快速记录。
5. 晚上完成 App 内日记与 AI 日终复盘。

10.3 失败与补救
1. 系统发现未达标。
2. 允许用户选择：失败、最低完成、补做中。
3. AI 输出失败原因诊断与一个窄而实用的修正建议。

10.4 纸质日记监督
1. 隔天一次提醒用户完成纸质日记。
2. 用户写完后在 App 内确认。
3. 若未完成，次日引导进入复盘，识别原因并调整。

11. 提醒系统设计

11.1 提醒层级
- 预备提醒：例如 22:15 开始收尾，为 23:00 睡眠目标做准备
- 截止提醒：例如 22:45 距离睡眠目标只剩 15 分钟
- 追踪提醒：例如 23:08 已错过目标，请诚实记录失败或进入补救模式

11.2 Hard / Soft 区别
- Soft：提醒更少、语气更柔和、更早展示降级完成选项
- Hard：提醒更密、语气更明确、不会无声滑过

12. 完成状态设计

所有习惯和任务都应收敛到以下状态之一：
- 完成（Complete）
- 最低完成（Minimum Complete）
- 失败（Missed）
- 补做（Made Up）
- 特殊免责（Excused，仅极少数情况）

其中，side project 默认不应设置太宽松的最低完成，因为你的核心诉求是“看得见的推进”。

13. AI 功能设计

13.1 AI Setup Assistant
将模糊目标转成可执行规则，包括周期、提醒时间、完成标准、最低标准和证据类型。

13.2 AI Failure Diagnosis
识别失败原因，如：
- 规则定义模糊
- 任务过大
- 提醒过晚
- 情绪阻力
- 精力崩盘
- 日程冲突

13.3 AI Coach
平时提供清晰建议，对关键项给出更坚定的督促。风格上应清楚、诚实、不油腻、不过度鸡汤。

13.4 AI Weekly Review
总结哪些习惯真的推动了产出，哪些规则设计有问题，哪里存在重复失败模式。

14. 游戏化机制

游戏化的目的不是娱乐，而是制造“可感知的代价与进步”。
- Control Score：综合当天整体掌控度
- Stability Meter：强调早睡、阅读、日记等基础稳定性
- Builder Rank：强调 side project 的持续输出
- 连胜与恢复机制：允许少量恢复，但不抹去真实失败记录

视觉和文案要成熟、克制、偏高级，不做卡通化奖章系统。

15. 信息架构

- Today：今日控制台
- Coach：AI 教练
- Journal：App 内日记 + 纸质日记监督
- Progress：趋势、得分、复盘
- Settings：规则与提醒管理

16. 首页结构

1. 顶部：日期、Control Score、AI 一句话提示
2. Protect Today：今日关键习惯
3. Push Today：今日关键项目动作
4. Hard commitments：其他高压项目
5. Soft commitments：其他柔性项目
6. Risk strip：AI 风险提示与模式预警

17. 视觉方向

推荐使用一种偏“私人控制室 / 编辑部指挥台”的语言：
- 不是可爱型效率工具
- 不是冷冰冰的数据后台
- 而是温暖、克制、坚定、略带压力感的高级界面

18. 风险与待决策问题

- 如果 Hard 项目设置太多，可能造成逃避感。
- 如果 AI 不够理解历史行为，会显得重复和表面化。
- 纸质日记未来是否需要拍照证明，V1 可先不做。
- 睡眠数据 V1 是否只靠自报，后续是否接入 Apple Health，可后续再定。

19. 开发优先级建议

第 1 阶段
- Onboarding
- Habit / Mission 数据模型
- Today 首页
- 基础提醒系统

第 2 阶段
- 完成状态系统
- App 内日记
- 纸质日记监督
- Progress 仪表盘

第 3 阶段
- AI 设置助手
- AI 失败诊断
- AI 日终教练与周复盘

20. 北极星定义

Life Console 帮助用户在一天结束时明确感受到：
我守住了底线，我推动了重要事情，而且我知道今天为什么成功或失败。
